AUC-ROC vs AUC-PR vs F1: ¿Qué métrica usar cuando tus clases están desbalanceadas?

Introducción

Cuando entrenamos un modelo de clasificación binaria, la primera métrica que solemos mirar es el accuracy (exactitud). Sin embargo, cuando las clases están desbalanceadas —por ejemplo, 99% de transacciones legítimas y 1% de fraudes— un modelo que prediga “legítimo” para todo tendría 99% de accuracy sin detectar ni un solo fraude.

Este artículo analiza tres métricas que sí capturan el rendimiento real en escenarios desbalanceados:

  • AUC-ROC: Área bajo la curva ROC
  • AUC-PR: Área bajo la curva Precision-Recall
  • F1 Score: Media armónica de precisión y recall

Usaremos un dataset sintético inspirado en detección de fraude para ilustrar por qué cada métrica cuenta una historia diferente.

Conceptos fundamentales

La matriz de confusión

Antes de hablar de curvas, es útil recordar los cuatro bloques que las alimentan:

Predicho Positivo Predicho Negativo
Real Positivo TP (Verdadero +) FN (Falso -)
Real Negativo FP (Falso +) TN (Verdadero -)

De aquí derivan todas las métricas:

\[\text{Recall (Sensibilidad)} = \frac{TP}{TP + FN}\]

\[\text{Precisión} = \frac{TP}{TP + FP}\]

\[\text{FPR (False Positive Rate)} = \frac{FP}{FP + TN}\]

AUC-ROC

La curva ROC enfrenta TPR (Recall) contra FPR a distintos umbrales de clasificación.

\[\text{TPR} = \frac{TP}{TP + FN}, \quad \text{FPR} = \frac{FP}{FP + TN}\]

El AUC-ROC es el área bajo esa curva. Tiene una interpretación probabilística elegante:

Es la probabilidad de que, dado un positivo y un negativo al azar, el modelo asigne mayor score al positivo.

El problema con clases desbalanceadas: El FPR incluye TN en el denominador. Cuando hay muchos negativos, el FPR permanece artificialmente bajo incluso con muchos falsos positivos. Esto hace que la curva ROC se vea optimista cuando en realidad el modelo falla en detectar la clase minoritaria.

AUC-PR

La curva PR enfrenta Precisión contra Recall a distintos umbrales.

\[\text{Precisión} = \frac{TP}{TP + FP}, \quad \text{Recall} = \frac{TP}{TP + FN}\]

La ventaja clave: no usa TN. En escenarios muy desbalanceados, esto significa que el rendimiento sobre la clase minoritaria no puede inflarse artificialmente.

El baseline de un clasificador aleatorio en PR-AUC es la prevalencia de la clase positiva (por ejemplo, 0.01 para un 1% de positivos), no 0.5 como en ROC. Esto hace la métrica más exigente y honesta.

F1 Score

El F1 es la media armónica entre precisión y recall:

\[F1 = 2 \cdot \frac{\text{Precisión} \times \text{Recall}}{\text{Precisión} + \text{Recall}} = \frac{2 \cdot TP}{2 \cdot TP + FP + FN}\]

Es una métrica a umbral fijo (normalmente 0.5), lo que la hace práctica pero menos completa que las curvas. Para datos desbalanceados se prefiere calcular por clase y promediar (macro o weighted).

Generamos un dataset con 10,000 muestras y un desbalance 50:1 (aproximadamente 2% de fraudes), similar al dataset real de tarjetas de crédito de Kaggle.

Entrenamiento de modelos

Entrenamos tres modelos distintos para comparar cómo las métricas los diferencian:

  1. Regresión Logística — modelo base, calibrado
  2. Random Forest — más potente, potencialmente mejor en la minoría
  3. Clasificador “optimista” — un modelo que siempre predice alta probabilidad (simula el problema de accuracy alto pero métricas de minoría malas)
Code 
# Modelo 1: Regresión Logística
lr = LogisticRegression(class_weight="balanced", max_iter=1000, random_state=42)
lr.fit(X_train, y_train)
lr_proba = lr.predict_proba(X_test)[:, 1]

# Modelo 2: Random Forest
rf = RandomForestClassifier(n_estimators=200, class_weight="balanced", random_state=42)
rf.fit(X_train, y_train)
rf_proba = rf.predict_proba(X_test)[:, 1]

# Modelo 3: "Ingenuo" — siempre predice probabilidad proporcional a la clase mayoritaria
# Simula un clasificador que aprende la distribución pero ignora patrones
naive_proba = np.full(len(y_test), y_train.mean()) + np.random.normal(0, 0.05, len(y_test))
naive_proba = np.clip(naive_proba, 0, 1)

models = {
    "Regresión Logística": lr_proba,
    "Random Forest": rf_proba,
    "Clasificador Ingenuo": naive_proba,
}

¿Por qué ROC puede engañar con datos desbalanceados?

Simulemos explícitamente el efecto con cuatro niveles de desbalance (1:1, 5:1, 20:1 y 100:1), entrenando el mismo modelo en cada caso.

Code 
# Generar datos y entrenar modelos para cada ratio
ratios = [1, 5, 20, 100]
sim_results = []

for ratio in ratios:
    n_pos_sim = 200
    n_neg_sim = n_pos_sim * ratio
    X_sim, y_sim = make_classification(
        n_samples=n_pos_sim + n_neg_sim,
        n_features=10, n_informative=5,
        weights=[n_neg_sim / (n_pos_sim + n_neg_sim), n_pos_sim / (n_pos_sim + n_neg_sim)],
        random_state=42,
    )
    Xtr, Xte, ytr, yte = train_test_split(X_sim, y_sim, test_size=0.3, stratify=y_sim, random_state=42)
    clf = RandomForestClassifier(n_estimators=100, random_state=42)
    clf.fit(Xtr, ytr)
    proba_sim = clf.predict_proba(Xte)[:, 1]
    sim_results.append((ratio, yte, proba_sim))

A medida que el ratio de desbalance aumenta:

  • ROC-AUC permanece artificialmente alto (aparenta buen rendimiento porque el FPR usa los abundantes TN)
  • PR-AUC cae notablemente y su baseline también se desplaza hacia abajo, revelando cuánto cuesta detectar la clase rara

El umbral como palanca: optimizar la métrica que importa

AUC-ROC y AUC-PR son métricas independientes del umbral — resumen el rendimiento a lo largo de todos los puntos de corte posibles. Pero en producción siempre hay que elegir un umbral concreto.

La pregunta clave es: ¿qué métrica quieres maximizar? La respuesta depende del negocio:

  • Maximizar Recall → no perder ningún fraude (aunque generes falsas alarmas)
  • Maximizar Precisión → solo alertar cuando estés muy seguro (aunque pierdas algunos fraudes)
  • Maximizar F1 → equilibrio entre ambos

El siguiente gráfico muestra, para el modelo Random Forest, cómo evolucionan Precisión, Recall y F1 al barrer todos los umbrales posibles. Las líneas verticales marcan el umbral óptimo para cada métrica.

Tip

¿Cómo leer este gráfico?

  • Recall (azul) decrece monotónicamente: umbrales bajos capturan más fraudes pero con mucho ruido.
  • Precisión (rojo) crece con el umbral: al exigir más confianza, cuando alertas casi siempre aciertas, pero te pierdes muchos fraudes.
  • F1 (violeta) tiene un máximo claro: es el punto de equilibrio entre los dos.

El umbral por defecto (0.5) raramente es el óptimo en datasets desbalanceados.

Guía de decisión: ¿cuál métrica usar?

Situación Métrica recomendada Razón
0 Clases balanceadas (<4:1) AUC-ROC TN abundantes no distorsionan el FPR
1 Desbalance moderado (4-10:1) AUC-ROC + AUC-PR Ambas dan perspectiva complementaria
2 Desbalance severo (>10:1) AUC-PR ROC infla el rendimiento real
3 Costo alto de falsos positivos Precisión / AUC-PR AUC-PR castiga FP sin contar TN
4 Costo alto de falsos negativos Recall / AUC-PR Recall nunca ignora FN
5 Necesitas un solo número F1 (macro o weighted) Resumen compacto de precisión y recall
6 Optimización de hiperparámetros AUC-PR o F1 Diferenciación clara entre modelos
7 Producción en tiempo real F1 o Precision@K Umbral explícito y controlable

Resumen

Métrica Qué mide Incluye TN Fiable con desbalance
AUC-ROC TPR vs FPR en todos los umbrales Sí No (puede inflar)
AUC-PR Precisión vs Recall en todos los umbrales No Sí
F1 Precisión y Recall en un umbral fijo No Sí (con cuidado)

Regla práctica:

Si tu dataset tiene más de 10:1 de desbalance, reporta siempre AUC-PR. Si puedes, muestra también el F1 óptimo (ajustando el umbral) y explica el tradeoff precisión-recall al stakeholder.

## Introducción

Cuando entrenamos un modelo de clasificación binaria, la primera métrica que solemos mirar es el **accuracy** (exactitud). Sin embargo, cuando las clases están desbalanceadas —por ejemplo, 99% de transacciones legítimas y 1% de fraudes— un modelo que prediga "legítimo" para todo tendría 99% de accuracy sin detectar ni un solo fraude.

Este artículo analiza tres métricas que sí capturan el rendimiento real en escenarios desbalanceados:

- **AUC-ROC**: Área bajo la curva ROC
- **AUC-PR**: Área bajo la curva Precision-Recall
- **F1 Score**: Media armónica de precisión y recall

Usaremos un dataset sintético inspirado en detección de fraude para ilustrar por qué cada métrica cuenta una historia diferente.

---

## Conceptos fundamentales

### La matriz de confusión

Antes de hablar de curvas, es útil recordar los cuatro bloques que las alimentan:

|                    | Predicho Positivo | Predicho Negativo |
|--------------------|:-----------------:|:-----------------:|
| **Real Positivo**  | TP (Verdadero +)  | FN (Falso -)      |
| **Real Negativo**  | FP (Falso +)      | TN (Verdadero -)  |

De aquí derivan todas las métricas:

$$\text{Recall (Sensibilidad)} = \frac{TP}{TP + FN}$$

$$\text{Precisión} = \frac{TP}{TP + FP}$$

$$\text{FPR (False Positive Rate)} = \frac{FP}{FP + TN}$$

### AUC-ROC

La curva **ROC** enfrenta TPR (Recall) contra FPR a distintos umbrales de clasificación.

$$\text{TPR} = \frac{TP}{TP + FN}, \quad \text{FPR} = \frac{FP}{FP + TN}$$

El **AUC-ROC** es el área bajo esa curva. Tiene una interpretación probabilística elegante:

> Es la probabilidad de que, dado un positivo y un negativo al azar, el modelo asigne mayor score al positivo.

**El problema con clases desbalanceadas:** El FPR incluye TN en el denominador. Cuando hay muchos negativos, el FPR permanece artificialmente bajo incluso con muchos falsos positivos. Esto hace que la curva ROC se vea optimista cuando en realidad el modelo falla en detectar la clase minoritaria.

### AUC-PR

La curva **PR** enfrenta Precisión contra Recall a distintos umbrales.

$$\text{Precisión} = \frac{TP}{TP + FP}, \quad \text{Recall} = \frac{TP}{TP + FN}$$

La ventaja clave: **no usa TN**. En escenarios muy desbalanceados, esto significa que el rendimiento sobre la clase minoritaria no puede inflarse artificialmente.

El baseline de un clasificador aleatorio en PR-AUC es la prevalencia de la clase positiva (por ejemplo, 0.01 para un 1% de positivos), no 0.5 como en ROC. Esto hace la métrica más exigente y honesta.

### F1 Score

El **F1** es la media armónica entre precisión y recall:

$$F1 = 2 \cdot \frac{\text{Precisión} \times \text{Recall}}{\text{Precisión} + \text{Recall}} = \frac{2 \cdot TP}{2 \cdot TP + FP + FN}$$

Es una métrica a **umbral fijo** (normalmente 0.5), lo que la hace práctica pero menos completa que las curvas. Para datos desbalanceados se prefiere calcular por clase y promediar (macro o weighted).

## Entrenamiento de modelos

Entrenamos tres modelos distintos para comparar cómo las métricas los diferencian:

1. **Regresión Logística** — modelo base, calibrado
2. **Random Forest** — más potente, potencialmente mejor en la minoría
3. **Clasificador "optimista"** — un modelo que siempre predice alta probabilidad (simula el problema de accuracy alto pero métricas de minoría malas)

```{python}
# Modelo 1: Regresión Logística

rf = RandomForestClassifier(n_estimators=200, class_weight="balanced", random_state=42)

rf_proba = rf.predict_proba(X_test)[:, 1]

# Modelo 3: "Ingenuo" — siempre predice probabilidad proporcional a la clase mayoritaria
```

## ¿Por qué ROC puede engañar con datos desbalanceados?

Simulemos explícitamente el efecto con cuatro niveles de desbalance (1:1, 5:1, 20:1 y 100:1), entrenando el mismo modelo en cada caso.

```{python}
# Generar datos y entrenar modelos para cada ratio
ratios = [1, 5, 20, 100]
sim_results = []

for ratio in ratios:
    n_pos_sim = 200
    n_neg_sim = n_pos_sim * ratio
    X_sim, y_sim = make_classification(
        n_samples=n_pos_sim + n_neg_sim,
        n_features=10, n_informative=5,
        weights=[n_neg_sim / (n_pos_sim + n_neg_sim), n_pos_sim / (n_pos_sim + n_neg_sim)],
        random_state=42,
    )
    Xtr, Xte, ytr, yte = train_test_split(X_sim, y_sim, test_size=0.3, stratify=y_sim, random_state=42)
    clf = RandomForestClassifier(n_estimators=100, random_state=42)
    clf.fit(Xtr, ytr)
    proba_sim = clf.predict_proba(Xte)[:, 1]
    sim_results.append((ratio, yte, proba_sim))
```

## El umbral como palanca: optimizar la métrica que importa

AUC-ROC y AUC-PR son métricas **independientes del umbral** — resumen el rendimiento a lo largo de todos los puntos de corte posibles. Pero en producción siempre hay que elegir **un umbral concreto**.

La pregunta clave es: ¿qué métrica quieres maximizar? La respuesta depende del negocio:

- **Maximizar Recall** → no perder ningún fraude (aunque generes falsas alarmas)
- **Maximizar Precisión** → solo alertar cuando estés muy seguro (aunque pierdas algunos fraudes)
- **Maximizar F1** → equilibrio entre ambos

El siguiente gráfico muestra, para el modelo Random Forest, cómo evolucionan Precisión, Recall y F1 al barrer todos los umbrales posibles. Las líneas verticales marcan el umbral óptimo para cada métrica.

```{python}
#| fig-width: 6.5
#| fig-height: 4.5
thresholds = np.linspace(0.005, 0.995, 400)

precisions, recalls, f1s = [], [], []
for t in thresholds:
    preds_t = (rf_proba >= t).astype(int)
    precisions.append(f1_score(y_test, preds_t, pos_label=1, average=None, zero_division=0)[1]
                      if preds_t.sum() > 0 else 0)
    recalls.append(f1_score(y_test, preds_t, pos_label=1, average=None, zero_division=0)[0]
                   if True else 0)
    f1s.append(f1_score(y_test, preds_t, average="binary", zero_division=0))

# Recalcular correctamente con sklearn precision/recall directo
from sklearn.metrics import precision_score, recall_score
precisions, recalls, f1s = [], [], []
for t in thresholds:
    preds_t = (rf_proba >= t).astype(int)
    precisions.append(precision_score(y_test, preds_t, zero_division=0))
    recalls.append(recall_score(y_test, preds_t, zero_division=0))
    f1s.append(f1_score(y_test, preds_t, zero_division=0))

precisions = np.array(precisions)
recalls    = np.array(recalls)
f1s        = np.array(f1s)

# Umbrales óptimos
t_best_f1   = thresholds[np.argmax(f1s)]
t_best_rec  = thresholds[np.argmax(recalls >= 0.90)]   # primer umbral con recall >= 90%
t_best_prec = thresholds[np.argmax(precisions >= 0.80)] # primer umbral con precision >= 80%

fig, ax = plt.subplots(figsize=(6.5, 4.5))

ax.plot(thresholds, precisions, color="#DC2626", lw=2.5, label="Precisión")
ax.plot(thresholds, recalls,    color="#2563EB", lw=2.5, label="Recall")
ax.plot(thresholds, f1s,        color="#7C3AED", lw=2.5, label="F1 Score")

# Marcadores de óptimos
opts = [
    (t_best_f1,   f1s[np.argmax(f1s)],          "#7C3AED", "F1 máx",       "top"),
    (t_best_rec,  recalls[np.argmax(recalls >= 0.90)], "#2563EB", "Recall ≥ 0.90", "bottom"),
    (t_best_prec, precisions[np.argmax(precisions >= 0.80)], "#DC2626", "Prec ≥ 0.80", "top"),
]

for t_opt, val, color, label, va in opts:
    ax.axvline(t_opt, color=color, linestyle="--", lw=1.5, alpha=0.7)
    ax.scatter([t_opt], [val], color=color, s=80, zorder=5)
    ax.annotate(
        f"{label}\nt={t_opt:.2f}",
        xy=(t_opt, val), xytext=(t_opt + 0.03, val + (0.05 if va == "top" else -0.1)),
        fontsize=8, color=color,
        arrowprops=dict(arrowstyle="->", color=color, lw=1),
    )

ax.axvline(0.5, color="grey", linestyle=":", lw=1.2, alpha=0.6)
ax.text(0.51, 0.05, "umbral\ndefault 0.5", fontsize=7.5, color="grey")

ax.set_xlabel("Umbral de clasificación", fontsize=11)
ax.set_ylabel("Valor de la métrica", fontsize=11)
ax.set_title("Precisión, Recall y F1 en función del umbral — Random Forest", fontsize=12)
ax.legend(fontsize=10, loc="center right")
ax.set_xlim([0, 1])
ax.set_ylim([0, 1.05])
plt.tight_layout()
plt.show()
```

::: {.callout-tip}
**¿Cómo leer este gráfico?**

- **Recall** (azul) decrece monotónicamente: umbrales bajos capturan más fraudes pero con mucho ruido.
- **Precisión** (rojo) crece con el umbral: al exigir más confianza, cuando alertas casi siempre aciertas, pero te pierdes muchos fraudes.
- **F1** (violeta) tiene un máximo claro: es el punto de equilibrio entre los dos.

El umbral por defecto (0.5) raramente es el óptimo en datasets desbalanceados.
:::

## Resumen

| Métrica | Qué mide | Incluye TN | Fiable con desbalance |
|---------|----------|:----------:|:---------------------:|
| AUC-ROC | TPR vs FPR en todos los umbrales | Sí | No (puede inflar) |
| AUC-PR  | Precisión vs Recall en todos los umbrales | No | Sí |
| F1      | Precisión y Recall en un umbral fijo | No | Sí (con cuidado) |

**Regla práctica:**

> Si tu dataset tiene más de 10:1 de desbalance, reporta siempre **AUC-PR**. Si puedes, muestra también el **F1 óptimo** (ajustando el umbral) y explica el tradeoff precisión-recall al stakeholder.
